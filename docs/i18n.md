# Internationalization in azulzin

How locales, translation, currency, and localized email work. azulzin is bilingual: **`pt-BR` is the default** (Brazil is the first public) and **`en-US` is fully supported**. Every user-facing string is a translation key — see the i18n rule in [CLAUDE.md](../CLAUDE.md) and the decision in [ADR 0006](decisions/0006-internationalization.md).

## Locales & config

| Locale | Role | Notes |
|---|---|---|
| `:"pt-BR"` | **Default** + UI choice | App strings under the `pt-BR:` key; formats/currency from `rails-i18n`. |
| `:"en-US"` | UI choice | Real locale (`rails-i18n` ships `en-US.yml`); inherits English app strings from `:en` via dash-split fallback. |
| `:en` | Fallback base only | Loaded so `en-US` overrides and `pt-BR` fallbacks resolve. **Never** shown as a UI option. |

```ruby
# config/application.rb (inside the Application class)
config.i18n.default_locale    = :"pt-BR"
config.i18n.available_locales = [:"pt-BR", :"en-US", :en]  # :en is the fallback base
config.i18n.fallbacks         = [:en]                      # NEVER `true` (would fall back to pt-BR)
# config.time_zone is separate from locale — set it if you want Brasília timestamps.
```

The **UI-visible** set (narrower than `available_locales`, which includes the internal `:en`) is declared once:

```ruby
# config/initializers/locale.rb
Rails.application.config.x.supported_locales = {
  "pt-BR" => "Português (Brasil)",
  "en-US" => "English (US)"
}
```

Gems (`Gemfile`): `rails-i18n ~> 8.1` (locale data), `http_accept_language ~> 2.1` (guest browser negotiation), and `i18n-tasks ~> 1.1` (development/test — key linting).

## How the active locale is chosen

Every request flows through an `around_action` that wraps the action in `I18n.with_locale` (per-request; a bare `I18n.locale =` leaks across requests on Puma). Resolution order, **whitelisted** against `supported_locales`, defaulting to `pt-BR`:

1. `params[:locale]` — a one-shot switch link
2. `session[:locale]` — sticky guest override
3. `current_user&.locale` — the logged-in member's saved preference (`users.locale`)
4. `Accept-Language` — guest browser preference (`http_accept_language`'s `compatible_language_from`, which only ever returns a listed locale)

```ruby
# app/controllers/application_controller.rb
around_action :switch_locale

private

def switch_locale(&action)
  I18n.with_locale(resolve_locale, &action)
end

def resolve_locale
  supported = Rails.application.config.x.supported_locales.keys        # %w[pt-BR en-US]
  candidate = params[:locale] || session[:locale] || current_user&.locale
  candidate = browser_preferred_locale if candidate.blank?
  supported.include?(candidate.to_s) ? candidate : I18n.default_locale # whitelist — mandatory
end

def browser_preferred_locale
  http_accept_language.compatible_language_from(
    Rails.application.config.x.supported_locales.keys
  )
end
```

**Switching** (writes the preference; no locale in URLs). Because Turbo Drive does not re-render `<html lang>` on in-place visits, the switcher forces a full navigation:

```ruby
# config/routes.rb
resource :locale, only: :update            # PATCH /locale

# app/controllers/locales_controller.rb
class LocalesController < ApplicationController
  def update
    loc = params.require(:locale)
    if Rails.application.config.x.supported_locales.key?(loc)
      current_user&.update(locale: loc)
      session[:locale] = loc
    end
    redirect_back fallback_location: root_path
  end
end
```

```erb
<%# language switcher — full navigation so <html lang> updates %>
<% Rails.application.config.x.supported_locales.each do |code, label| %>
  <%= button_to label, locale_path, method: :patch, params: { locale: code },
        data: { turbo: false }, class: "btn btn-ghost btn-xs" %>
<% end %>
```

The layout sets the document language from the (already BCP-47) locale:

```erb
<html lang="<%= I18n.locale %>">   <%# "pt-BR" or "en-US" %>
```

## Writing translations

- **Views/partials:** lazy lookup `t(".title")` → `<controller>.<action>.title`.
- **Controllers/flash:** `flash[:notice] = t(".created")` (lazy, scoped to `<controller>.<action>`).
- **Models:** put names/attributes/errors under the framework namespaces so validations translate for free:

```yaml
# config/locales/pt-BR.yml
pt-BR:
  activerecord:
    models: { user: "usuário" }
    attributes:
      user: { email_address: "e-mail", password: "senha" }
    errors:
      models:
        user:
          attributes:
            email_address: { taken: "já está em uso" }
```

File layout: two flat files, `config/locales/pt-BR.yml` (`pt-BR:`) and `config/locales/en.yml` (`en:` — the shared English base `:"en-US"` inherits). Split by domain only when a file gets unwieldy.

## Money, numbers & dates

Store money as **integer cents** (`bigint amount_cents`); format at the edge with `BigDecimal` + `number_to_currency`, which reads the symbol/format from the active locale:

```erb
<%= number_to_currency(amount_cents.to_d / 100) %>
<%# pt-BR → "R$ 1.234,56"   en-US → "$1,234.56" %>
```

Rules: never hardcode `R$`/`$`; convert cents with `.to_d / 100` (not `/ 100.0` — floats corrupt money); when parsing pt-BR input (`"1.234,56"`), normalize the `.`/`,` separators before `to_d` (never `to_f`). Locale governs *format* only — it does not convert BRL↔USD; hold an explicit `currency` code if you ever store more than one.

## Localized email

Mailers set the locale from the **recipient's** stored preference inside an `around_action`, so email language is correct for both `deliver_now` and `deliver_later` regardless of the enqueuing request's locale:

```ruby
# app/mailers/application_mailer.rb
class ApplicationMailer < ActionMailer::Base
  default from: "no-reply@azulzin.com.br"
  around_action :set_locale

  private

  def set_locale(&block)
    I18n.with_locale(params&.[](:user)&.locale || I18n.default_locale, &block)
  end
end

# app/mailers/user_mailer.rb — subject comes from the locale file
class UserMailer < ApplicationMailer
  def email_verification
    @user = params[:user]
    mail(to: @user.email_address, subject: default_i18n_subject)
  end
end

# call site
UserMailer.with(user: @user).email_verification.deliver_later
```

```yaml
# config/locales/pt-BR.yml
pt-BR:
  user_mailer:
    email_verification: { subject: "Confirme seu e-mail no azulzin" }
    # PasswordsMailer (generator) subject keys go here too
```

One view per mailer with `t(".…")` inside — no per-locale view files (no email has a different layout per language).

## Hotwire/Turbo

Nothing special: every Turbo Drive/Frame/Stream request is a normal HTTP request through `ApplicationController#switch_locale`, so the locale is applied on each response — no JS, no client state. The single caveat is the stale `<html lang>` after an in-place switch, handled by the full-navigation switcher above.

## Keeping locales in sync (CI)

`i18n-tasks` finds missing/unused keys and interpolation mismatches. Gate CI with a test:

```ruby
# test/i18n_test.rb
require "test_helper"
require "i18n/tasks"

class I18nTest < ActiveSupport::TestCase
  setup { @i18n = I18n::Tasks::BaseTask.new }

  test "no missing keys" { assert_empty @i18n.missing_keys.values.flatten, "run: i18n-tasks missing" }
  test "no unused keys"  { assert_empty @i18n.unused_keys.values.flatten,  "run: i18n-tasks unused" }
end
```

Handy: `bundle exec i18n-tasks health | missing | unused | normalize`. Assert on resolved keys (`I18n.t("…", locale: :"pt-BR")`) rather than hardcoded copy so tests survive wording changes.
